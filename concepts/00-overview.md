# 00-overview.md - 六大核心概念总览

> 来源：OpenAI 2026-02-11，作者 Ryan Lopopolo  
> 背景：3 人团队用 Codex 从空仓库到 100 万行代码，5 个月，零手写代码

## 🎯 什么是 Harness Engineering？

**Harness Engineering（驭缰工程）** 是一种新的工程范式：

> **工程师不再写代码，而是设计环境、明确意图、构建反馈回路，让 AI 智能体可靠地完成工作。**

### 范式对比

| 传统模式 | Harness 模式 |
|---------|-------------|
| 人类写代码 → 机器运行 | 人类设计约束 → 智能体写代码 → 机器运行 |
| 工程师的输出 = 代码 | 工程师的输出 = 约束系统（AGENTS.md、架构规则、linter、反馈回路） |
| 人工 Code Review | 机械化验证（lint/test/build 自动通过） |
| PR 周期长（天/周） | PR 周期短（分钟/小时） |

### 核心数据

- 3 人团队 → 5 个月 → ~100 万行代码 → ~1500 个 PR
- 人均每天 3.5 个 PR
- 扩展到 7 人后吞吐量仍在增长
- 单次 Codex 运行可持续 6+ 小时（通常在人类睡眠时间）

## 🔑 六大核心概念

### 1. Repo as Source of Truth（代码库即真理）

**核心**：所有规则必须可机械化验证，不能依赖"人类理解"。

**为什么重要**：
- 智能体无法理解模糊的"最佳实践"
- 只有可执行的规则才能被强制执行
- 代码库本身必须成为唯一的事实来源

**如何落地**：
- 用 `AGENTS.md` 作为智能体导航入口
- 所有架构决策必须写成可验证的规则
- 禁止"凭感觉"的代码，一切以 CI 通过为准

**参考文档**：`01-repo-as-source-of-truth.md`

---

### 2. Mechanical Enforcement（机械化执行）

**核心**：用 linter、测试、CI/CD 强制执行约束，不依赖人工审查。

**为什么重要**：
- 人类会疲劳、会疏忽
- 智能体需要明确的"通过/失败"信号
- 只有机械化才能规模化

**如何落地**：
- 自定义 linter 检查架构规则
- 预提交钩子（pre-commit hooks）拦截违规
- CI 流水线自动运行所有检查
- 测试覆盖率作为合并门槛

**参考文档**：`02-mechanical-enforcement.md`

---

### 3. Entropy and Garbage Collection（熵增与垃圾回收）

**核心**：系统会自然熵增，需要定期清理"技术债务垃圾"。

**为什么重要**：
- 智能体会产生大量临时代码
- 规则会随时间失效
- 不清理会导致系统崩溃

**如何落地**：
- 定期运行"垃圾回收"脚本
- 标记并删除过期配置
- 自动化检测"僵尸代码"
- 建立"技术债务看板"

**参考文档**：`03-entropy-and-garbage-collection.md`

---

### 4. Agent Readability（智能体可读性）

**核心**：代码首先给智能体看，其次给人看。

**为什么重要**：
- 智能体是主要"读者"和"修改者"
- 人类可读 ≠ 智能体可读
- 命名、结构、注释都要考虑智能体理解

**如何落地**：
- 使用明确的命名（避免缩写、歧义）
- 文件结构遵循固定分层（Types → Config → Repo → Service → Runtime → UI）
- 注释解释"为什么"而非"是什么"
- 避免智能体难以理解的"技巧性代码"

**参考文档**：`04-agent-readability.md`

---

### 5. Throughput Changes Merge（吞吐量优先合并）

**核心**：PR 生命周期要短，偶发失败通过重跑解决。

**为什么重要**：
- 智能体吞吐量远超人类注意力
- 等待人工 Review 是瓶颈
- 偶发测试失败不应阻塞合并

**如何落地**：
- 自动化测试重跑机制
- 短周期 PR（< 1 小时）
- 人类只审查"关键变更"
- 建立"信任但验证"的文化

**参考文档**：`05-throughput-changes-merge.md`

## 🗺️ 学习路径

### Phase 1：理解核心概念

1. 阅读本文件（概览）
2. 逐篇研读六大概念文档
3. 在 `thinking/` 中写下质疑和延伸

### Phase 2：形成个人观点

1. 质疑每个概念的适用边界
2. 结合自身经验提出改进
3. 记录到 `thinking/` 目录

### Phase 3：实战演练

1. 选择 `practice/` 中的小项目
2. 用 Harness 范式从零构建
3. 记录遇到的问题和解决方案

### Phase 4：反馈迭代

1. 将踩坑经验整理到 `feedback/`
2. 更新概念文档补充新发现
3. 优化工具和流程

### Phase 5：输出作品

1. 提炼成文章、工具或教程
2. 发布到 `works/` 目录
3. 分享给社区

## 🚨 常见误区

❌ **误区 1**：把 Harness Engineering 当成"完全不用写代码"  
✅ **正解**：工程师从"写代码"转向"设计约束系统"，仍需深度参与架构设计

❌ **误区 2**：认为智能体可以完全自治  
✅ **正解**：人类仍是"掌舵者"，负责明确意图、设定边界、验证结果

❌ **误区 3**：忽视熵增，认为一次设置就能永久有效  
✅ **正解**：必须建立定期"垃圾回收"机制，持续清理技术债务

❌ **误区 4**：用人类可读的标准要求智能体可读性  
✅ **正解**：智能体需要更明确、更结构化的信息，人类可读是"附加价值"
